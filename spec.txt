1. Product Overview
What VenueOS is, who it serves, and why it exists
1.1 Product Summary
VenueOS is a full-stack, AI-powered crowd lifecycle management platform for large-scale sporting events. It addresses the entire timeline of an attendee's event day — from pre-event planning by organisers, through real-time crowd monitoring, to controlled exit after the match.
The platform operates across two distinct user interfaces: an Organiser Dashboard for venue staff, and an Attendee Mobile Web App for ticket holders. Both interfaces share a common Firestore database and are coordinated by Gemini AI, which acts as the intelligent planning and operations engine.
Primary context: IPL and international cricket matches at large Indian venues such as MA Chidambaram (Chennai), Wankhede (Mumbai), and Eden Gardens (Kolkata), each hosting 40,000 to 80,000 attendees.

2. Functional Requirements
Every feature the system must deliver, grouped by module
Module A — GIS Road & Gate Planner (Organiser)
Purpose: Allow organisers to define the venue map, mark entry/exit gates, map roads to gates, and use Gemini to generate an optimal road-blocking and crowd routing plan before the event.
FR-A1  Venue map setup   [Must Have]
Organiser can create a venue profile by entering venue name, city, and capacity. The system embeds a Google Map centred on the venue. Organiser can place polygon markers for the venue boundary, gate markers (labelled Gate 1, Gate 2... Gate N), road entry markers on surrounding roads, and amenity markers (water, restroom, first aid, food stall).
FR-A2  Gate-to-road assignment   [Must Have]
For each gate, the organiser can assign one or more approach roads from the road markers placed on the map. The UI shows a list of all gates with a multi-select dropdown of roads. Each road-gate pair is saved to Firestore.
FR-A3  Perimeter definition   [Must Have]
Organiser draws a circular or polygonal geo-fence perimeter around the venue on the map. The perimeter radius is saved. This perimeter is used for attendee slot validation on event day.
FR-A4  Gemini road-blocking plan generation   [Must Have]
Organiser inputs: expected crowd size, event start time, and number of gates. Pressing 'Generate Plan' calls the Gemini API with venue data, gate-road assignments, and crowd size. Gemini returns a structured plan: which roads to block, which roads to keep open, recommended traffic diversion instructions per gate, and staff deployment positions. The plan is rendered as a text report and as colour-coded overlays on the map.
FR-A5  Plan export   [Must Have]
The generated plan can be exported as a PDF or copied as plain text for distribution to ground teams.
Module B — Batch & Slot Scheduler (Organiser)
Purpose: Allow organisers to divide all attendees into batches of 100-200 people, assign each batch a gathering point and time slot for entry and exit, and use Gemini to compute this automatically.
FR-B1  Batch generation via Gemini   [Must Have]
Organiser inputs: total attendee count, number of gates, event duration, number of outer gathering zones available, and transport capacity per trip. Pressing 'Generate Batches' calls Gemini which computes: number of batches, batch size per gate, entry time window per batch (spaced to prevent simultaneous gate rushes), exit time window per batch. Gemini outputs a structured JSON schedule.
FR-B2  Batch code assignment   [Must Have]
Each batch is assigned a unique code (e.g. B-A1, B-A2, B-B1...). The first letter indicates the gate (A = Gate 1, B = Gate 2), the number indicates the sequence. Codes are stored in Firestore under the venue event document.
FR-B3  Gathering point assignment   [Must Have]
Each batch is assigned to one outer gathering zone. Gathering zones are locations placed on the map by the organiser (FR-A1 extension). Each zone can hold a maximum of N attendees (configurable). The system distributes batches across zones evenly.
FR-B4  Entry and exit slot display   [Must Have]
The organiser dashboard shows a full schedule table: Batch Code | Gate | Gathering Zone | Entry Window | Exit Window | Transport Departure Time. The schedule is editable — organisers can manually adjust time windows and save.
FR-B5  Attendee slot notification   [Must Have]
When the schedule is finalised, the system marks it as 'Published'. Attendees who have authenticated and booked their slot receive their batch code, gathering zone address, entry window, and exit window in their app.
Module C — Outer Gathering Zone & Transport System (Organiser + Attendee)
Purpose: Manage the outer gathering zones where attendees assemble by batch before being transported to the venue. The gathering zone is deliberately placed at an open, spacious location away from the venue perimeter to avoid pre-event crowding near the stadium.
FR-C1  Gathering zone map setup   [Must Have]
Organiser places gathering zone markers on the map at locations that are: (a) a minimum of 500m from the venue perimeter, (b) have sufficient open space for the batch size. Each zone has a name, GPS coordinates, and maximum capacity. Zones are visible to attendees on their live map.
FR-C2  Transport fleet setup   [Must Have]
Organiser registers available transport units (buggies, buses, or shuttles) with: transport ID, type, capacity, and assigned route (gathering zone to venue gate). This enables the dispatch dashboard.
FR-C3  Batch transport dispatch   [Must Have]
When a batch's entry window is approaching (configurable threshold, default 20 minutes), the organiser dashboard shows a 'Dispatch' button for that batch. Clicking it: marks the batch as 'In Transit' in Firestore, notifies attendees in that batch that their transport is ready, and records dispatch timestamp.
FR-C4  QR scan at transport boarding   [Must Have]
Each transport boarding point has a staff-operated QR scanner (browser-based, using the device camera). When an attendee scans their QR pass, the system validates: (a) batch code matches the dispatched batch, (b) attendee is not already marked as boarded. On success, the system marks the attendee as 'Boarded' in Firestore. Invalid scans show an error with the reason.
FR-C5  Attendee navigation to gathering zone   [Must Have]
The attendee app shows turn-by-turn directions from the attendee's current GPS location to their assigned gathering zone using Google Maps Directions API (walking or driving mode selectable). Navigation activates automatically when the attendee opens the app after slot booking.
Module D — Geo-fenced Perimeter Control (Organiser + Attendee)
Purpose: Enforce slot-based entry by detecting when an attendee enters the venue perimeter and validating their entry slot before allowing them to proceed to the gate.
FR-D1  Perimeter detection on attendee app   [Must Have]
When the attendee app is open and the attendee's GPS location enters the perimeter drawn by the organiser (FR-A3), the app checks: (a) Is the attendee's batch entry window currently active? If yes: show green status 'You are cleared to enter. Proceed to [Gate X].' If no: show amber warning 'You have arrived early. Your entry window is [time]. Please wait at [Gathering Zone name].' and display navigation back to the gathering zone.
FR-D2  QR scan at perimeter gate   [Must Have]
A staff-operated QR scanner at the perimeter boundary validates: (a) batch entry window is active, (b) attendee is marked as 'Boarded' (came via transport), (c) QR pass is not expired or revoked. On success, marks attendee as 'Perimeter Cleared' in Firestore.
FR-D3  Early arrival redirect   [Must Have]
If an attendee's GPS enters the perimeter before their entry window, the app automatically shows a push notification and in-app alert with navigation directions back to their assigned gathering zone and their correct entry time.
Module E — QR Pass System (Attendee + All Checkpoints)
Purpose: A single QR code governs all attendee checkpoints. The QR pass is generated only after slot booking is confirmed and contains all information needed at every stage.
FR-E1  QR pass generation   [Must Have]
After an attendee completes slot booking (selects entry window and exit window), the system calls a Cloud Function that generates a QR pass. The QR encodes a signed JSON payload containing: attendee ID, event ID, batch code, gate assignment, gathering zone ID, entry window start/end timestamps, exit window start/end timestamps, and a HMAC signature for tamper detection. The QR is displayed in the attendee app and can be saved to the device camera roll.
FR-E2  QR scanner (staff)   [Must Have]
Any staff member can open the scanner view on their phone browser. The scanner uses the device camera via the jsQR library. On scan, it decodes the payload, verifies the HMAC signature, and checks the appropriate checkpoint rule (transport boarding, perimeter, gate entry, exit). Results are shown instantly with a green/red colour indicator and written to Firestore.
FR-E3  Checkpoint stages   [Must Have]
The QR pass tracks six checkpoint stages in order: (1) Slot Booked, (2) Transport Boarded, (3) Perimeter Cleared, (4) Gate Entered, (5) Inside Venue, (6) Exited. Each stage is timestamped in Firestore. Attempting to scan a stage out of order shows an error.
FR-E4  Exit slot enforcement   [Must Have]
On exit, the scanner checks that the attendee's exit window is currently active. If the attendee tries to exit before their window, the scanner shows an amber warning with their exit window time. In an emergency, organisers can override from the dashboard.
Module F — Live Crowd Intelligence Dashboard (Organiser)
Purpose: Real-time crowd density visualisation and Gemini-powered anomaly detection for organisers and staff.
FR-F1  Crowd density heatmap   [Must Have]
The organiser dashboard shows a Google Maps heatmap layer over the venue and surroundings. Density data is sourced from: (a) GPS pings from attendee app (sent every 30 seconds while app is open), (b) QR scan events at each checkpoint (each scan adds a density point at the scan location). The heatmap updates every 60 seconds. Intensity zones: Low (green), Medium (amber), High (red), Critical (dark red).
FR-F2  Per-gate crowd counter   [Must Have]
The dashboard shows a live count card for each gate: people currently in queue (estimated from GPS pings within 50m of gate), people entered in last 10 minutes (from gate scan events), percentage of gate capacity used.
FR-F3  Gemini anomaly detection   [Must Have]
Every 60 seconds, a Cloud Run scheduled job assembles a density snapshot (headcount per zone, rate of change per zone, match state, time to next natural break) and calls Gemini API. Gemini analyses the snapshot and returns: predicted congestion zones in the next 5-15 minutes, confidence level, recommended action for each zone (redirect attendees, deploy more staff, open additional gate), and natural language summary. The dashboard shows these predictions as highlighted zones with tooltips.
FR-F4  Auto-drafted staff instructions   [Must Have]
When Gemini predicts a critical congestion event (confidence > 70%), the system auto-generates a staff instruction card on the dashboard: 'Send 3 additional staff to Gate 3. Announce redirect to Gate 7.' Organisers can one-click approve or edit before sending.
FR-F5  Attendee redirection push   [Must Have]
When an organiser approves a redirect instruction, the system sends an in-app notification to all attendees currently within 200m of the congested gate, directing them to the alternative gate.
Module G — Attendee Mobile Web App
Purpose: The attendee-facing interface. Accessible only after ticket scan + OTP authentication. Not publicly accessible.
FR-G1  Ticket scan and OTP authentication   [Must Have]
The attendee visits the VenueOS web app URL (or scans a QR on their physical ticket). The app prompts them to enter their registered mobile number. Firebase Auth sends an OTP SMS. On verification, the attendee is authenticated and their profile is loaded from Firestore (matched by phone number to the ticketing database).
FR-G2  Slot booking screen   [Must Have]
After authentication, the attendee selects an entry time slot from available windows (slots not yet at capacity). They also select an exit time slot. On confirmation, the system assigns them a batch code and gathering zone, generates their QR pass (FR-E1), and saves all data to Firestore.
FR-G3  Home screen / dashboard   [Must Have]
Post-booking home screen shows: QR pass (large, tappable to expand), batch code and gathering zone name, entry window time, exit window time, current checkpoint stage with progress indicator, and a live countdown to their entry window if not yet active.
FR-G4  Navigation screen   [Must Have]
Tap 'Navigate' to get Google Maps turn-by-turn directions to the gathering zone (pre-entry) or to the assigned gate (once inside perimeter). Uses browser Geolocation API for current position and Google Maps Directions API for routing.
FR-G5  Live venue map   [Must Have]
Shows a full-screen Google Map of the venue with: all gate markers, amenity markers (water, restroom, first aid, food stall), gathering zones, the perimeter boundary, and a simplified crowd density overlay (colour-coded zones). Attendee's own location shown as a blue dot.
FR-G6  Amenity finder   [Must Have]
A bottom sheet UI lists all amenities by category (water, restrooms, food, first aid). Tapping an amenity centres the map on it and shows walking distance. A 'Report Issue' button opens the incident reporting form pre-filled with the amenity location.
FR-G7  Incident and amenity reporting   [Must Have]
The report form has two types: (a) General incident: description text field + optional photo/video upload (max 30 seconds video, Cloud Storage). (b) Amenity issue: amenity pre-selected from map, issue type dropdown (broken, out of stock, unclean, other), optional photo. On submit, Gemini API categorises the report, assigns a priority level, and writes to Firestore with routing to the relevant staff role. Attendee sees a confirmation with a report reference number.
FR-G8  Notifications   [Must Have]
The app shows in-app banners for: entry window approaching (30 minutes before), transport dispatched (immediate), crowd redirect advisory (immediate), exit window approaching (30 minutes before). If browser push notifications are enabled, these also appear as system notifications.
Module H — Authentication & Access Control
FR-H1  Organiser authentication   [Must Have]
Organisers log in with Google Sign-In (Firebase Auth with Google provider). Only pre-approved Google accounts (listed in a Firestore 'admins' collection) can access the organiser dashboard. Unauthorised accounts see an access denied screen.
FR-H2  Attendee authentication   [Must Have]
Phone OTP via Firebase Auth. The attendee app is not publicly accessible — all routes redirect to the auth screen if not authenticated. Session persists across page reloads using Firebase's session persistence.
FR-H3  Staff scanner access   [Must Have]
Staff access the QR scanner via a direct URL with a staff token (short-lived JWT generated by organisers from the dashboard). No full account needed — the token grants scanner-only access for a defined event day.

3.6 QR Pass — Payload Schema & Signing
PAYLOAD SCHEMA
{ "v": 1, "aid": "attendee_uid", "eid": "event_id", "bc": "batch_code", "gid": "gate_id", "zid": "zone_id", "ews": 1713456000, "ewe": 1713459600, "xws": 1713492000, "xwe": 1713495600, "sig": "hmac_sha256_hex" }
Field key: v=version, aid=attendeeId, eid=eventId, bc=batchCode, gid=gateId, zid=zoneId, ews=entryWindowStart (Unix timestamp), ewe=entryWindowEnd, xws=exitWindowStart, xwe=exitWindowEnd, sig=HMAC-SHA256 signature.
SIGNING
Signing key = HMAC_SECRET env variable (set in Cloud Run). Sign the JSON string of all fields except 'sig' using HMAC-SHA256. The scanner validates by re-computing the HMAC and comparing to the sig field. Any mismatch = invalid/tampered QR.
QR FORMAT
Encode the entire JSON payload (including sig) as a base64 string, then encode that as a QR code using qrcode.js. Error correction level: H (highest). QR code size: 300x300px minimum for reliable scanning.
3.7 Perimeter Detection Logic
Implement this logic in the attendee app JavaScript:
ALGORITHM
1. Every 30 seconds, call navigator.geolocation.getCurrentPosition(). 2. Compute distance from current position to perimeterCenter using the Haversine formula. 3. If distance <= perimeterRadiusMeters: fetch attendee's entryWindowStart and entryWindowEnd from Firestore. 4. If current time is within entry window: show green 'Cleared to enter' UI. 5. If current time is before entry window: show amber warning with remaining time and navigation back to gathering zone. 6. If attendee is outside perimeter: normal UI, show navigation to gathering zone.
HAVERSINE
function haversineDistanceMeters(lat1, lng1, lat2, lng2) { const R = 6371000; const dLat = (lat2-lat1)*Math.PI/180; const dLng = (lng2-lng1)*Math.PI/180; const a = Math.sin(dLat/2)**2 + Math.cos(lat1*Math.PI/180)*Math.cos(lat2*Math.PI/180)*Math.sin(dLng/2)**2; return R * 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1-a)); }
4. Process Flows
Step-by-step flows for every critical path in the system
4.1 Pre-Event Organiser Planning Flow
This flow is completed by the organiser in the days before the event.
Organiser logs in with Google account (Firebase Auth). System checks Firestore 'admins' collection — access denied if not listed.
Organiser creates a new event: enters venue name, city, date, start time, and total capacity.
Venue map screen loads with Google Maps centred on venue. Organiser places: venue boundary polygon, gate markers (one per gate), road approach markers on surrounding streets, amenity markers for water/restrooms/food/first aid.
Organiser assigns roads to gates: for each gate, selects which road markers lead to it.
Organiser draws the perimeter circle: clicks 'Set Perimeter', clicks centre point on map, drags to set radius. Radius is shown in metres. Saves.
Organiser places outer gathering zone markers: at least one per gate, minimum 500m from perimeter. Sets zone name and max capacity.
Organiser registers transport fleet: adds each buggy/bus with ID, type, capacity, and route.
Organiser clicks 'Generate Road Plan'. System calls Prompt 1 via Gemini API. Response rendered as map overlays + text report. Organiser reviews and saves.
Organiser clicks 'Generate Batch Schedule'. Enters total attendees and transport capacity. System calls Prompt 2 via Gemini API. Full schedule table rendered. Organiser reviews, adjusts if needed, saves.
Organiser clicks 'Publish Schedule'. System sets schedulePublished = true. All authenticated attendees can now see their batch assignments and begin slot booking.
4.2 Attendee Pre-Entry Flow
This flow is completed by the attendee before arriving at the venue.
Attendee navigates to VenueOS URL (or scans QR on physical ticket). App loads the auth screen.
Attendee enters their registered mobile number. Firebase Auth sends OTP SMS.
Attendee enters OTP. Firebase Auth verifies. System looks up attendee record in Firestore by phone number. If not found, prompts attendee to verify ticket ID to create profile.
Home screen loads. If schedule is not yet published, attendee sees a waiting screen with event date/time. If published, attendee sees slot selection screen.
Attendee selects entry slot from available windows. Attendee selects exit slot. Confirms booking.
System assigns batch code, gate, and gathering zone. Calls Cloud Function to generate signed QR pass payload. Writes all data to Firestore attendee document.
Home screen updates: shows QR pass, batch code, gathering zone name and address, entry window countdown, and a 'Navigate' button.
On day of event, attendee taps 'Navigate' to get directions to their gathering zone. App uses Google Maps Directions API for routing.
4.3 Event Day Entry Flow (Complete)
This is the full end-to-end flow on event day from gathering zone to seat.
Attendee arrives at outer gathering zone as per their batch code. The zone is a large open space 500m+ from the venue.
Organiser's dashboard shows all batches and their status. When a batch's entry window is 20 minutes away, a 'Dispatch Transport' button activates.
Organiser clicks 'Dispatch'. System marks batch as 'In Transit'. Push notification sent to all attendees in that batch: 'Your transport is ready at [Zone Name]. Board now.'
At the gathering zone, a staff member operates the QR scanner on their phone. Each attendee scans their QR pass. System validates batch code and marks attendee as 'Boarded'. Capacity count updates in real time on the organiser dashboard.
Transport departs when batch is fully boarded (or at scheduled departure time, whichever comes first).
Transport arrives at venue perimeter. As attendees exit transport, attendee app detects entry into geo-fence. App shows green 'Cleared to enter. Proceed to [Gate X].'
At perimeter gate: staff QR scanner validates (a) entry window active, (b) attendee boarded, (c) signature valid. Marks attendee as 'Perimeter Cleared'. Any invalid scan shows error reason.
Attendee walks to their assigned gate. At gate: staff QR scanner does final entry scan. Marks attendee as 'Gate Entered' and 'Inside'. Crowd count for that gate updates.
Inside venue, attendee uses live map to find amenities, see crowd density zones, and navigate to seat section.
4.4 Real-Time Density Analysis Flow (Runs Every 60 Seconds)
Cloud Scheduler triggers the density analysis Cloud Function every 60 seconds during the event window (2 hours before start to 1 hour after end).
Cloud Function queries Firestore: (a) gps_pings in last 2 minutes grouped by zone (using geohash proximity), (b) scan_events in last 10 minutes grouped by checkpointId, (c) current attendee checkpoint stage counts.
For each gate zone, computes: current headcount (GPS pings within 50m), rate of change vs previous snapshot, percentage of gate capacity used.
Assembles density snapshot JSON. Fetches current match state (can be manually entered by organiser or read from a match data field in the event document).
Calls Gemini API with Prompt 3. Parses the JSON response.
Writes new density_snapshot document to Firestore. Organiser dashboard listens via Firestore onSnapshot and updates heatmap automatically.
For any prediction with confidence > 0.7 and severity 'high' or 'critical': creates an alert document in Firestore with status 'pending'. Dashboard shows alert card with approve/edit/dismiss options.
When organiser approves an alert: Cloud Function sends in-app notification to all attendees within the target radius of the congested zone.
4.5 Incident Report Flow
Attendee taps 'Report Issue' from live map or amenity finder.
Selects report type: General Incident or Amenity Issue. If Amenity Issue, amenity is pre-populated from map tap.
Enters description. Optionally uploads photo or short video (max 30s). Media uploads to Cloud Storage. Upload URL stored temporarily.
Taps Submit. Backend API endpoint /attendee/:uid/report receives the report with all fields plus attendee's current GPS location.
Backend calls Gemini API with Prompt 4. Gemini returns category, priority, routedToRole, and actionRequired.
Backend writes complete incident_reports document to Firestore with Gemini classification fields populated.
Organiser dashboard shows new report card in the incidents panel, highlighted by priority colour. Staff of the routed role see the report in their staff view.
Attendee sees confirmation: 'Report #[ID] received. Priority: [level]. Estimated response: [time].'
4.6 Post-Match Exit Flow
30 minutes before estimated match end, organiser activates 'Exit Mode' on dashboard.
System sends push notifications to all attendees: 'Your exit window is [time]. Please make your way to [Exit Gate X] when notified.'
Exit windows are staggered by batch — same logic as entry. Batches are notified when their exit window is active.
Attendee navigates to exit gate using live map. At exit gate, staff QR scanner validates exit window is active and marks attendee as 'Exited'.
If attendee tries to exit before their window, scanner shows amber warning: 'Exit window: [time]. Please wait in designated waiting area.'
Organiser can override any exit restriction for individual attendees from the dashboard (emergency override).
Transport returns from venue to outer gathering zones for attendees who came by the transport system.
