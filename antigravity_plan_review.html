Errors that must be fixed before building
✗
Phase 4 typo — "HMRC-SHA256" should be "HMAC-SHA256" Error
The plan description for qr.py says "HMRC-SHA256 signing". HMRC is the UK tax authority. The correct algorithm is HMAC-SHA256 as specified in Section 3.6. If this wording is passed to Antigravity as a prompt, it may generate incorrect or hallucinated code. Fix this before Phase 4 prompt.
✗
QR checkpoint stage order is incomplete Error
The plan lists the QR stage flow as: "Slot Booked → Boarded → Perimeter → Inside → Exited" — missing the "Gate Entered" stage. The spec (FR-E3) defines 6 stages: Slot Booked → Transport Boarded → Perimeter Cleared → Gate Entered → Inside Venue → Exited. Gate Entered and Inside are two separate checkpoints. If qr.py is built with 5 stages, the gate entry scan will have no stage to write to and the validator will break.
✗
Frontend CSS mentions "glassmorphism" — conflicts with mobile performance requirement Risk
The plan proposes "glassmorphism elements" for the organiser dashboard CSS. Glassmorphism uses backdrop-filter: blur(), which is GPU-intensive and can cause significant lag on budget Android phones that attendees will use on event day. It is fine on the desktop organiser dashboard but must not be used in frontend-attendee/css/style.css. Explicitly instruct Antigravity to use flat, performant CSS for the attendee app only.
Items from the spec missing from the plan
!
No mention of Cloud Scheduler setup for density analysis Missing
The spec (Section 3.1 and 4.4) requires Cloud Scheduler to trigger the density_analysis Cloud Function every 60 seconds during the event window. The plan lists functions/main.py as an HTTP-triggered function but does not include the Cloud Scheduler job creation in deploy.sh or any setup instructions. Without this, the density heatmap never auto-updates — a judge will see a static map.
!
Firestore TTL policy for gps_pings not mentioned Missing
The spec (Section 3.3) requires a Firestore TTL policy on the gps_pings collection using the expireAt field, so old pings auto-delete after 4 hours. The plan lists firestore.indexes.json and firestore.rules but does not mention TTL setup. Without this, the gps_pings collection will grow unboundedly during the event and slow down density queries.
!
Staff scanner token generation (FR-H3) not in any phase Missing
The spec defines a staff scanner access system: organisers generate short-lived JWT tokens from the dashboard, and staff access /scanner?token=[staff_token] without a full account. The plan lists scanner.py as a router but does not include the token generation endpoint in organiser.py or the token verification middleware. Without this, the scanner has no access control — anyone with the URL can scan.
!
Firestore seeding / demo data not planned Missing
The spec (Section 8.2) specifies exactly what needs to be pre-seeded in Firestore for the demo: one event (CSK vs MI), 8 gates, 4 gathering zones, 20+ amenities, 200+ batches, 3-5 test attendees, simulated density snapshots, and one pre-classified incident report. The plan has no seed script or data setup phase. With only 3 days left, this needs to be scheduled — without seeded data there is nothing to demo.
!
generate_qr_on_booking Cloud Function not in Phase 8 Missing
The spec defines two Cloud Functions: density_analysis (included) and generate_qr_on_booking (not mentioned). The generate_qr_on_booking function triggers on a Firestore write when an attendee's checkpointStage becomes 'booked' and writes the QR payload back to the attendee document. Without it, the QR pass is never generated after slot booking — the most important attendee flow breaks completely.
!
Offline resilience for QR scanner not planned Missing
The spec (Screen O6) requires the scanner to cache the last 100 validated QR payloads locally and queue scan events when offline, syncing when reconnected. Stadium network is unreliable with 50,000 people. The plan makes no mention of this. It is acceptable to defer this for the hackathon, but Antigravity should be explicitly told to add localStorage caching of HMAC-validated payloads in the scanner JS.
